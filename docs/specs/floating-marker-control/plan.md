> The code is the terrain and this plan is a map: where they disagree, the
> terrain wins. A divergence at the decision level flows back to spec.md
> instead of being worked around.

## Approach

- `OverlayApp`이 현재 도구와 설정값의 소유권을 계속 갖고, `Marker`는 도구 전환·인스펙터 노출·배치만 담당한다.
- `Marker`의 내부 패널 상태를 `닫힘 / 펜 / 텍스트`로 단순화한다.
  - 비활성 도구 첫 클릭: 해당 도구로 전환만 한다.
  - 활성 도구 재클릭: 해당 도구의 인스펙터를 열거나 닫는다.
  - 칠판은 도구 선택과 독립적으로 토글하되 열린 인스펙터는 닫는다.
- 펜과 텍스트를 동일한 깊이의 도구 버튼으로 표현하고, 인스펙터에서만 속성을 하위 깊이로 보여준다.
  - 펜: 색상 5개와 굵기 5개를 두 행으로 제공한다.
  - 텍스트: 숫자 대신 비례형 `T` 샘플 5개를 제공한다.
  - 실제 `24/32/44/60/80px` 값은 접근성 이름과 툴팁으로 확인할 수 있게 한다.
  - 접힌 `T`는 현재 크기에 따라 `15/17/19/22/24px`로 변하고, 인스펙터 표본은 비교가 잘 되는 `13/16/20/25/30px`를 사용한다.
- 인스펙터는 화면 경계에서 6px 간격을 유지하도록 좌우 위치를 보정하고, 화살표는 활성 도구 중앙을 계속 가리키도록 독립적으로 이동시킨다.
- 경계 계산은 렌더링과 분리된 결정 로직으로 만들어 양쪽 경계와 화살표 위치를 결정론적으로 테스트한다.
- 기존 색상·굵기 상수와 `OverlayApp` 콜백 계약은 재사용한다. 텍스트 크기는 `TextSizeKey` 키와 설정 스키마를 유지하면서 실제 값과 기본값만 새 스케일로 바꾼다.

검토했지만 채택하지 않는 접근:

- 인스펙터 상태를 `OverlayApp`까지 끌어올리는 방식은 화면 표시 상태를 앱 상태와 불필요하게 결합한다.
- 범용 팝오버나 디자인 시스템 모듈을 새로 만드는 방식은 현재 단일 사용처에 비해 변경 범위가 크다.
- CSS 위치 지정만 사용하는 방식은 패널을 경계 안으로 옮기면서 화살표를 도구 중앙에 유지하는 동작을 안정적으로 검증하기 어렵다.

## Order

1. `Marker` 단위 테스트와 텍스트 편집 통합 테스트를 승인된 상호작용 기준으로 먼저 변경한다.
   - 비활성 도구의 첫 클릭과 활성 도구의 재클릭을 구분한다.
   - 기존 “편집 중 T 클릭으로 편집 종료” 테스트는 “편집 중 펜 클릭으로 편집 종료”로 교체한다.
   - 편집 중 활성 T를 누르면 텍스트 인스펙터가 열리고 초안은 유지되는지 검증한다.
2. `Marker`를 세 개의 고정 컨트롤—펜, 텍스트, 칠판—구조로 재구성한다.
   - 펜과 텍스트는 상호 배타적인 선택 상태를 갖는다.
   - 활성 상태는 중립 배경으로 통일한다.
   - 펜 아이콘은 선택된 잉크 색상을 사용하는 자유곡선 형태로 표시한다.
   - 접힌 `T`는 현재 텍스트 크기의 compact scale을 반영한다.
   - 기존 4px 드래그 임계값과 위치 저장 동작은 유지한다.
3. 문맥형 인스펙터를 구현한다.
   - 펜 인스펙터는 색상과 굵기를 한 패널에 함께 제공한다.
   - 텍스트 인스펙터는 비례형 `T` 샘플을 제공하고 선택 즉시 실제 텍스트 크기를 적용한 뒤 닫는다.
   - 현재 선택값, 행 레이블, 도구 이름을 스크린 리더가 구분할 수 있게 한다.
4. 경계 배치와 화살표 정렬을 적용한다.
   - 좌우 화면 초과를 보정한다.
   - 상단 공간이 부족하면 독 아래로 연다.
   - 패널이 이동해도 화살표는 펜 또는 텍스트 버튼 중앙을 가리킨다.
5. `OverlayApp`, 다국어 문자열, 기존 설정 저장 흐름과 통합한 뒤 전체 회귀 검증을 수행한다.

## Acceptance criteria

- [ ] 접힌 독에는 펜, 텍스트, 구분선, 칠판만 표시된다.
- [ ] 펜과 텍스트의 크기·배경·선택 표현이 동일한 위계를 갖는다.
- [ ] 비활성 도구를 처음 클릭하면 도구만 전환되고 인스펙터는 열리지 않는다.
- [ ] 활성 도구를 다시 클릭하면 해당 인스펙터가 열린다.
- [ ] 펜 인스펙터에서 색상과 굵기를 한 번에 선택할 수 있다.
- [ ] 텍스트 크기는 숫자가 아닌 비례형 `T` 샘플로 표시된다.
- [ ] 실제 텍스트 크기는 `24/32/44/60/80px`이며 첫 기본값은 `32px`다.
- [ ] 접힌 `T`는 `15/17/19/22/24px`, 인스펙터 표본은 `13/16/20/25/30px` 표시 스케일을 사용한다.
- [ ] 각 텍스트 크기의 접근성 이름 또는 툴팁에는 실제 px 값이 포함된다.
- [ ] 텍스트 편집 중 크기를 변경해도 편집 초안과 포커스가 유지된다.
- [ ] 텍스트 편집 중 펜을 선택하면 편집을 확정하고 펜 모드로 전환한다.
- [ ] `T` 단축키는 텍스트 모드만 전환하며 인스펙터를 자동으로 열지 않는다.
- [ ] 칠판 토글, 바깥 클릭, 캔버스 입력, 키보드 종료 동작은 열린 인스펙터를 닫는다.
- [ ] 인스펙터는 좌우 경계에서 최소 6px 안쪽에 위치한다.
- [ ] 경계 보정 후에도 화살표가 활성 도구 버튼을 가리킨다.
- [ ] 상단 공간이 부족하면 인스펙터가 독 아래에 열린다.
- [ ] 독 드래그, 위치 저장, 4px 클릭·드래그 구분이 회귀하지 않는다.

## Verification and seams

- `Marker` 단위 테스트: 도구 전환 상태, 인스펙터 토글, 선택값, 바깥 클릭, 칠판 토글, 좌우·상단 배치, 화살표 위치를 검증한다.
- 텍스트 편집 통합 테스트: 실제 `DrawingCanvas`와 `Marker` 조합에서 크기 변경, 펜 전환, 편집 초안 유지 여부를 검증한다.
- `OverlayApp` 테스트: 커서, 설정 저장, 칠판 명령, 텍스트 모드 상태가 유지되는지 확인한다.
- 다국어 테스트: 새 도구 이름, 속성 행 레이블, px 접근성 문자열의 한국어·영어 출력을 확인한다.
- 전체 검증: 프런트엔드 테스트, 의존성 경계 검사, 빌드, 프로젝트 전체 테스트 게이트를 실행한다.
- 네이티브 수동 검증: 실제 macOS 앱에서 `⌥Tab`으로 진입한 뒤 도구 전환, 텍스트 편집, 독 드래그, 우측·상단 경계 배치를 실제 마우스로 확인한다.

## Off-limits

- Rust, Tauri IPC, 설정 스키마 변경
- 펜 스트로크와 텍스트 렌더링 엔진 변경
- 실행 취소·다시 실행 및 두 단계 `Esc` 계약 변경
- 전역 단축키 변경
- 지우개·도형 등 새로운 도구 추가
- 승인된 스펙과 프로토타입의 제품 결정 변경
- 범용 디자인 시스템 또는 새 팝오버 프레임워크 도입

## Risks and open questions

- 패널과 화살표를 함께 보정하면 렌더링 직후 측정 순서에 따라 순간적인 위치 흔들림이 생길 수 있다. 위치 계산을 분리하고 한 번의 레이아웃 갱신으로 적용한다.
- 기존에는 편집 중 T 버튼이 편집을 종료했지만, 새 구조에서는 활성 T가 인스펙터를 연다. 이 마이그레이션을 통합 테스트로 명시적으로 고정한다.
- 비례형 샘플의 시각 크기와 실제 적용 px 값이 다르므로, 샘플 간 상대적 차이는 명확히 유지하고 실제 값은 접근성 이름과 툴팁에서 제공한다.
- 네이티브 `NSPanel`의 실제 포인터 전달은 브라우저 테스트만으로 증명할 수 없으므로 최종 수동 검증이 필요하다.
