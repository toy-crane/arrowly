> The code is the terrain and this plan is a map: where they disagree, the
> terrain wins. A divergence at the decision level flows back to spec.md
> instead of being worked around.

## Approach

- 현재 구현의 세 도구 위계, 비활성 도구 전환과 활성 도구 재클릭, 텍스트 편집 연동, 렌더링된 패널 측정 방식은 유지한다. 아직 남아 있는 이전 텍스트 크기표, 고정폭·레이블형 인스펙터, 고정 굵기 펜 아이콘과 고정 크기 `T`만 승인된 프로토타입에 맞춘다.
- `OverlayApp`은 현재 도구와 설정 저장을 계속 소유하고, `Marker`는 `닫힘 / 펜 / 텍스트` 패널 상태와 도구 전환만 관리한다.
- `Marker` 내부의 공통 패널 표면과 배치 책임을 overlay 도메인의 `ToolInspector`로 분리한다. 이 컴포넌트는 8px 패딩, 4px 행·항목 간격, 콘텐츠 기반 너비, 상하 방향 전환, 좌우 6px 클램프와 활성 도구를 가리키는 화살표를 소유하고 도구별 선택지는 자식으로 받는다.
- `FreehandToolLiveStrokeIcon`은 현재 색상과 `2/2.6/3.2/4.2/5.6px` 시각 굵기를, `LiveTextSizeIcon`은 현재 텍스트 단계의 `15/17/19/22/24px` 표시 크기를 받는 순수 표시 컴포넌트로 둔다.
- 텍스트의 5개 `TextSizeKey`는 유지하되 실제 값은 `24/32/44/60/80px`, 기본 키는 `small`(`32px`)로 바꾼다. 실제 출력 크기는 마커 크기, 펜 굵기, 화면 해상도와 완전히 독립된 고정 CSS px다.
- 접힌 `T`와 인스펙터의 `T` 표본은 실제 출력 크기를 계산하지 않는다. 같은 키를 각각 `15/17/19/22/24px`와 `13/16/20/25/30px` 표시표에 대응한다.
- 시각적 `색·굵기·크기` 레이블은 제거하고 패널, 행, 선택지의 지역화된 접근성 이름은 유지한다.

검토했지만 채택하지 않는 접근:

- 인스펙터 상태를 `OverlayApp`으로 올리는 방식은 일시적인 표시 상태를 앱 상태와 불필요하게 결합한다.
- 범용 디자인 시스템 팝오버를 만드는 방식은 현재 한 개 사용처에 비해 변경 범위가 크다.
- 실제 px를 아이콘과 표본에 직접 렌더링하는 방식은 42×32px 셀과 도구 위계를 깨뜨린다.
- 설정 키를 px 값으로 교체하는 방식은 settings.json과 Rust 저장 계약의 마이그레이션을 요구하므로 제외한다.

## Order

한 구현 세션에서 아래 네 개의 논리적 커밋으로 진행한다. 각 단계가 끝날 때 관련 테스트를 통과시켜 다음 단계의 기준점을 만든다.

1. 텍스트 크기 계약을 먼저 바꾼다.
   - 실제 크기를 `24/32/44/60/80px`, 기본 키를 `small`로 바꾼다.
   - 크기 증감, 레이아웃, 접근성 문자열, 설정 기본값 테스트를 새 값에 맞춘다.
   - 저장된 5단계 키는 그대로 유효하므로 스키마 마이그레이션 없이 새 크기표를 적용한다.
2. `ToolInspector`를 추출한다.
   - 공통 표면, 화살표, 실제 크기 측정, 좌우 클램프와 상단 반전을 `Marker`에서 옮긴다.
   - `Marker`는 기준 버튼과 접근성 이름, 도구별 콘텐츠만 전달한다.
   - 실제 패널 높이와 마커의 현재 top을 사용해 방향을 정하고 리사이즈·마커 이동 시 다시 계산한다.
3. 확정된 캡슐과 아이콘을 적용한다.
   - 독을 `외곽 8px + 펜 42px + 도구 간 4px + 텍스트 42px + 구분선 좌우 8px + 보드 42px`의 163px 구조로 맞춘다.
   - 자유곡선 아이콘을 승인된 단일 스윕 경로로 교체하고 색상·굵기 표시표를 연결한다.
   - 접힌 `T`와 인스펙터 표본에 서로 다른 compact scale을 연결한다.
   - 인스펙터의 시각적 레이블, 고정 최소 폭과 이전의 3px·5px 간격을 제거한다.
4. 상호작용과 통합 회귀를 고정한다.
   - 값 선택, 바깥 클릭, 캔버스 입력, 블랙보드와 키보드 도구 전환 시 패널을 닫는다.
   - 편집 중 활성 T는 패널을 열고 초안을 유지하며, 펜 선택은 편집을 확정하고 전환한다.
   - 전체 자동화 게이트를 통과한 뒤 실제 macOS `NSPanel`에서 수동 검증한다.

## Acceptance criteria

- [ ] 접힌 독에는 펜, 텍스트, 구분선, 블랙보드만 있고 전체 폭은 `163px`다.
- [ ] 펜과 텍스트 사이에는 `4px` 간격이 있고 두 도구의 크기·활성 표현은 같은 위계다.
- [ ] 자유곡선 아이콘은 현재 색상과 상대 굵기를 함께 보여주며 모든 단계에서 열린 단일 스윕을 유지한다.
- [ ] 펜 인스펙터에는 색상 5개와 굵기 5개가 레이블 없이 두 행으로 나타난다.
- [ ] 텍스트 인스펙터에는 숫자 대신 비례형 `T` 표본 5개가 나타난다.
- [ ] 실제 텍스트 크기는 `24/32/44/60/80px`, 첫 기본값은 `32px`다.
- [ ] 실제 텍스트 크기는 마커 크기, 펜 굵기와 화면 해상도 변화에 영향받지 않는다.
- [ ] 접힌 `T`는 `15/17/19/22/24px`, 인스펙터 표본은 `13/16/20/25/30px` 표시표를 사용한다.
- [ ] 시각적 레이블 없이도 패널·행·선택지의 접근성 이름으로 속성과 실제 값을 구분할 수 있다.
- [ ] 비활성 도구 첫 클릭은 전환만 하고 활성 도구 재클릭은 해당 인스펙터를 토글한다.
- [ ] 텍스트 편집 중 크기를 바꿔도 입력값, 캐럿과 편집 세션이 유지된다.
- [ ] 텍스트 편집 중 펜을 선택하면 현재 내용을 확정하고 펜 모드로 전환한다.
- [ ] 인스펙터는 좌우 경계에서 최소 6px 안쪽에 있고 화살표는 활성 도구를 가리킨다.
- [ ] 실제 패널 높이가 상단 공간보다 크면 인스펙터가 독 아래로 열린다.
- [ ] 독 드래그, 위치 복원·저장과 4px 클릭·드래그 구분이 회귀하지 않는다.

## Verification and seams

- 크기 상수 테스트에서 실제 값, 기본 키, 단계 증감과 양 끝의 no-op을 검증한다.
- `ToolInspector`의 배치 계산 seam에서 패널 폭·높이, 마커 위치와 뷰포트를 입력으로 좌우 클램프, 상단 반전과 화살표 위치를 검증한다.
- `Marker` 테스트에서 도구 전환, 패널 토글, 값 선택, 간격, 두 아이콘의 표시 매핑과 접근성 트리를 검증한다.
- 실제 `DrawingCanvas + Marker` 통합 테스트에서 편집 중 T 패널 열기, 크기 적용 후 초안·포커스 유지와 펜 전환을 검증한다.
- `OverlayApp` 테스트에서 기본값 로딩, 새 텍스트 크기 저장, 커서와 블랙보드 명령의 기존 배선을 확인한다.
- 자동화 게이트는 `bun run test:all`로 의존성 경계, 프런트엔드 커버리지, Rust 도메인 커버리지와 빌드를 함께 확인한다.
- 네이티브 검증은 `bun run tauri dev` 후 `⌥Tab`으로 진입해 실제 마우스로 도구 전환, 5색×5굵기, 한글 편집 중 크기 변경, 독 드래그와 좌우·상단 배치를 확인한다.

## Off-limits

- Rust, Tauri IPC와 settings.json 키 구조
- 펜 스트로크 및 텍스트 캔버스 렌더링 방식
- 실행 취소·다시 실행과 두 단계 `Esc` 계약
- 전역 단축키와 텍스트 편집의 `⌘+`·`⌘−` 계약
- 마커 위치 저장 형식과 4px 드래그 임계값
- 지우개·도형 등 새로운 도구
- 범용 팝오버 프레임워크 또는 shared 디자인 시스템
- 승인된 스펙과 프로토타입의 크기·간격·아이콘 결정

## Risks and open questions

- 기존에 저장된 `medium` 키는 실제 크기가 30px에서 44px로 바뀐다. 승인된 스펙대로 키는 유지하고 새 스케일을 적용하며 별도 마이그레이션은 하지 않는다.
- 컴포넌트 추출 후 첫 측정 전에 패널이 흔들릴 수 있다. 배치 계산과 DOM 스타일 적용을 한 레이아웃 사이클에서 수행하고 중간 위치 테스트로 고정한다.
- 자유곡선 아이콘의 경로·색·굵기는 자동화할 수 있지만 5색×5굵기의 미세한 시각 품질은 실제 44px 네이티브 캡슐에서 확인해야 한다.
- non-activating `NSPanel`의 합성 클릭과 실제 포인터 전달은 다를 수 있으므로 브라우저·jsdom 테스트만으로 완료 판정하지 않는다.
