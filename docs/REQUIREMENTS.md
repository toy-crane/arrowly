# Arrowly 요구사항 문서

## 제품 정의

화면 녹화 강의를 만들 때, 정지된 화면 위에 순간적으로 그리고 지우는 일회용 주석 오버레이.

두 가지 용도를 하나의 자유곡선 펜으로 해결한다.

- 화면 위 강조: 기존 화면(코드, 슬라이드, 브라우저) 위에 밑줄·동그라미·화살표를 얹는다.
- 자유곡선 다이어그램: 빈 화면 또는 화면 위에 원과 화살표를 손으로 그려 구조(예: 루프)를 설명한다.

### 정체성 경계

- 마크(한 번 긋고 못 고치며 전체와 함께 지워짐)만 다루면 주석기다. Arrowly는 여기에 머문다.
- 선택·이동·크기변경·저장이 되는 순간 다이어그램 편집기(Excalidraw)가 된다. 그 선을 넘지 않는다.

## 기술 스택

- 프레임워크: Tauri (Rust 백엔드 + OS 웹뷰)
- 프론트엔드: Vite + React + TypeScript
- 패키지 매니저: Bun
- 타깃: macOS (활성 모니터 1개)

## 기능 요구사항

### 그리기

- 자유곡선 펜 하나로 원·화살표를 포함해 모두 손으로 그린다.
- 색은 형광펜 세트처럼 5색으로 고정한다: 옐로우(#FFD400), 오렌지(#FF7A00), 핑크(#FF2D95), 그린(#2ED573), 블루(#00AEEF).
- 대비 외곽선은 두지 않는다. 채도 높은 원색만으로 대비를 만든다. 흰 배경 위 옐로우가 약한 건 실제 형광펜과 같은 한계로 두고, 안 보이면 다른 색으로 바꾼다.
- 굵기는 5단계로 고른다(색과 같은 개수로 확정). 기본값(중간 단계)은 화면 크기 비율 기준으로 잡는다. 거의 바꾸지 않으므로 그리는 도중 스크롤·키 실시간 조절은 두지 않는다(인지 부하).
- 색·굵기 변경은 플로팅 마커에서 한다(아래 "플로팅 마커" 참조).

### 교정

- ⌘Z로 마지막 획을 취소하고 ⌘⇧Z로 되살린다.
- 단, ⌘Z가 오버레이에 도달하려면 오버레이가 키 입력을 받아야 하는데, 그 과정에서 아래 앱의 활성 상태(메뉴바, 창 포커스 표시)를 뺏으면 녹화 화면이 죽어 보인다. 두 조건을 동시에 만족해야 한다: (a) ⌘Z가 획을 취소하고 (b) 아래 앱이 활성으로 보인다. non-activating panel(NSPanel) 방식을 우선 검토한다. **이것이 이 앱의 최우선 기술 리스크다.**
- 전체 지우기를 빠른 단축키로 제공한다. 새 설명을 시작할 때, 그리고 숨겨뒀던 그림이 낡았을 때 화면을 비우는 1급 동작이다.
- 지우개는 없다. 개별 요소 편집도 없다(append-only).

### 상태 모델

- 단축키가 그리기 모드와 통과 모드를 전환한다.
  - 그리기 ON: 오버레이가 마우스를 잡아 그린다.
  - 그리기 OFF: 오버레이가 화면에서 사라지고, 마우스가 아래 앱으로 통과한다.
- 통과 모드 전환은 그림의 "숨김"이지 "삭제"가 아니다 (확정). 다시 그리기 모드로 들어오면 이전 그림이 그대로 살아 있다. 삭제는 명시적 동작(⌘Z, 전체 지우기)으로만 일어난다. 예외: 직전과 다른 모니터에서 그리기가 켜지면 좌표가 무효이므로 전체 지운다.
- 통과 모드에서 그림을 화면에 계속 표시(Epic Pen 방식)하지는 않는다 — "그림을 띄운 채 아래 앱을 조작"은 비목표.
- Esc는 언제나 통과 모드로 나가는 예약 키다. 사용자 재설정 대상이 아니며, 커스텀 단축키가 실패해도 살아 있어야 하는 안전장치다(아래 "안전장치" 참조).

### 블랙보드

그리기 모드의 배경 상태다. 켜면 오버레이 배경이 불투명 검정이 되어 화면 전체가 빈 검은 판이 된다. 화면 위 강조가 아니라 자유곡선 다이어그램("빈 화면" 용도)을 위한 기능이다.

- 진입점은 셋이다: 플로팅 마커의 보드 버튼, ⌘B(그리기 모드 중), 메뉴바 메뉴의 체크 항목. 통과 모드에서 메뉴로 켜면 그리기 모드로 함께 진입한다.
- 획 버퍼는 화면 주석과 공유한다. 보드를 켜고 꺼도 획은 남는다 — 삭제는 명시적 동작(⌘Z, 전체 지우기)으로만 일어난다.
- 보드 상태는 그리기 OFF/ON을 넘나들며 유지된다(숨김≠삭제와 같은 규칙). 앱을 재시작하면 투명으로 리셋하며, 영속화하지 않는다.
- 창 레벨은 올리지 않는다. Dock과 메뉴바가 보드 위로 보일 수 있다 — 메뉴바 클릭 가능 안전장치가 우선이고, 전체화면 공간에서는 어차피 둘 다 숨는다.
- 화이트보드(흰 배경)는 두지 않는다. 잉크 5색은 검정 위에서만 전부 성립한다(옐로우는 흰 배경에서 죽는다).

### 단축키 등록 정책

전역 단축키는 등록된 동안 그 키 조합을 다른 모든 앱에서 가로챈다. 빼앗는 시간을 최소화한다.

- 그리기 토글: 상시 전역 등록. 아무 앱에서나 즉시 그리기 시작이 이 앱의 존재 이유이므로 유일한 상시 키다.
- 전체 지우기: 전역 등록하지 않는다. 통과 모드에선 그림이 숨겨져 있어 즉시 지울 일이 없고, 그리기 모드에선 오버레이가 키 입력을 받으므로 ⌘Z와 같은 경로(오버레이 keydown)로 처리한다.
- ⌘Z/⌘⇧Z: 전역 등록하지 않는다. 오버레이가 키 입력을 받는 구조(교정 항목의 non-activating panel)로 해결한다.
- 블랙보드 토글(⌘B): 전역 등록하지 않고 ⌘Z와 같은 경로(오버레이 keydown)로 처리한다. 재설정 대상이 아닌 고정 키다. 수식키 없는 단독 키는 쓰지 않는다 — 이후 텍스트 입력이 생기면 겹친다. ⌥+글자도 macOS에서 특수문자 입력이라 제외한다.
- 등록 실패(다른 전역 단축키와 충돌 등)는 조용히 넘기지 않고 UI로 알린다. 앱 내부 단축키(예: 특정 앱의 메뉴 단축키)와의 충돌은 원천 탐지가 불가능하므로, 흔치 않은 기본값과 사용자 재설정으로 완화한다.

### 플로팅 마커

색·굵기를 바꾸는 컨트롤이자 그리기 모드 표시를 겸한다. "떠 있는 툴바"가 아니라 "열 수 있는 상태등"에 가깝다 — 평소엔 작게, 조작할 때만 펼친다.

- 그리기 ON일 때 화면 구석에 가로로 작게 상주하고, 통과 모드에선 사라진다(떠 있음=그리기 ON). 드래그로 옮긴다. 기본 위치는 좌하단(디자인 시안 v1 확정).
- 높이 44px 캡슐(시안 v1의 34px에서 실사용 피드백으로 확대), 항상 다크 반투명 재질 — 어떤 화면 위에서도 동일하게 보이도록 라이트/다크를 따라가지 않는다.
- 평소 모습은 현재 색 점 + 굵기 단계 표시 + 블랙보드 버튼(칠판 글리프) 세 부분이다. 굵기 단계 표시는 세로획 5개(굵기 계단, 높이 균일)를 상시 그리고 현재 단계만 밝힌다 — 단계가 여럿 있음이 아이콘 자체에서 읽혀야 한다. 셀 3개는 고정폭·동일 패딩으로 통일한다. 글자 레이블은 쓰지 않는다.
- 색 부분을 탭하면 5색이, 굵기 부분을 탭하면 5단계가 **캡슐 위 팝오버**로 펼쳐진다(확정: 캡슐 자체는 위치·크기 불변이라 레이아웃이 예측 가능하다). 화면 최상단 근처에서는 아래로 뜬다. 팝오버는 화면 밖으로 나가지 않게 클램프한다.
- 펼침에서 현재 굵기는 채움으로 표시한다(현재만 꽉 참, 나머지는 테두리). 선택 표시에 색은 쓰지 않는다 — 색은 색 차원 전용이다. 색 펼침에서 현재 색은 중립색 링으로 표시한다. 블랙보드 버튼의 활성 표시도 중립 하이라이트다.
- 고르면 적용하며 닫힌다. 같은 부분을 다시 탭하거나, 바깥 탭·그리기 시작으로도 닫힌다.

### 화면 범위

- 활성화한(커서가 있는) 모니터 하나만 덮는다.
- 판정 시점: 그리기 토글 ON 순간의 커서 위치로 고정한다. 다른 모니터로 옮기려면 토글을 껐다 다시 켠다.
- 디스플레이 구성 변경(해상도 변경, 모니터 연결·해제, 잠자기 복귀, 클램셸 전환) 이벤트를 받아 오버레이를 재배치한다.

### 안전장치 (잠금 방지)

그리기 ON 상태의 오버레이는 화면 전체의 클릭을 삼킨다. 빠져나오는 길이 막히면 사용자는 자기 맥을 조작할 수 없다. 탈출구는 서로 독립적인 코드 경로로 여러 개 둔다.

- Esc = 무조건 통과 모드. 웹뷰가 아니라 Rust 쪽에서 처리해, 웹뷰가 얼어도 살아 있게 한다.
- 메뉴바 아이콘은 항상 클릭 가능해야 한다. 오버레이 창 레벨이 메뉴바를 덮지 않는지 초기에 검증한다(전체화면 앱 대응과 상충할 수 있음).
- 끄는 수단이 하나도 확보되지 않은 상태(토글 단축키 등록 실패 등)로는 그리기 ON에 진입하지 않고 이유를 알린다.

## 온보딩과 학습성

이 앱은 보이지 않고 모드가 있는 도구라, 발견성과 학습성이 핵심 문제다.

- 첫 실행 시 20초 인터랙티브 튜토리얼로 직접 그려보게 한다. 이 흐름 안에서 사용자가 단축키를 직접 설정한다.
- 온보딩 구성(디자인 시안 v1 확정): ① 직접 그려보기 ② 단축키 정하기 ③ Esc 탈출구·⌘Z 체험, 3단계로 끝. 마무리 전용 페이지는 두지 않고 메뉴바 상주 안내는 ③의 마지막 한 줄로 처리한다. 권한이 필요한 구현으로 확정되면 ②와 ③ 사이에 조건부 권한 안내 단계를 넣는다.
- 재설정 가능한 앱 고유 단축키는 두 개로 고정한다(그리기 토글, 전체 지우기). 나머지는 OS 표준(⌘Z 등)과 고정 키(Esc, 블랙보드 ⌘B)를 쓴다.
- 단축키는 사용자가 재설정할 수 있다. 좋은 기본값을 제공하되 변경할 수 있게 한다. 기본값(확정): 그리기 토글 ⌥Tab, 전체 지우기 ⌥⌫.
- 단축키 설정에는 키 조합을 직접 눌러 캡처하는 레코더 UI가 필요하다. 전역 충돌로 등록이 실패하면 그 자리에서 보여준다.
- 전체화면 앱 위에서는 메뉴바가 숨어 단축키가 유일한 진입로다. 온보딩에서 알려준다.
- 메뉴바 메뉴에 모든 동작과 그 단축키를 표시해 상시 치트시트가 되게 한다.
- 그리기 시작은 단축키뿐 아니라 메뉴바 클릭으로도 가능하게 한다.
- 그리기 모드 표시는 두 가지로 한다: 커서가 현재 색·굵기의 점으로 바뀌는 것, 그리고 플로팅 마커가 떠 있는 것. 통과 모드에선 커서가 평범한 화살표로 돌아오고 마커는 사라진다.
- 점 커서에는 이중 링을 두른다 — 또렷한 흰 링(어두운 화면용) + 어두운 헤어라인(밝은 화면용). 어떤 배경에서도 위치를 찾게 한다(커서 시안 A 확정). 획 자체에는 어떤 경우에도 외곽선을 넣지 않는다 — "대비 외곽선 금지"는 잉크에만 적용되는 원칙이다.
- 화면에 띄운 것은 녹화에 찍히므로 마커는 슬림하게 유지하고, 깨끗한 테이크가 필요하면 숨길 수 있게 한다.

## 앱 형태

- Dock과 앱 스위처에서 숨기고 메뉴바에만 상주한다. 예외(확정): 온보딩·설정 같은 일반 창이 열려 있는 동안만 Regular 정책으로 전환해 Dock에 표시한다 — 메뉴바 유틸의 표준 패턴으로, 앱 메뉴(⌘W 등)가 살아나고 마지막 창이 닫히면 다시 숨는다.
- 메뉴바에 종료 항목을 넣는다. 메뉴바 전용 앱은 표준 메뉴가 없어 ⌘Q가 기본 동작하지 않을 수 있으므로 종료 경로를 확인한다.
- 메뉴바 아이콘은 템플릿 이미지로 만들어 다크/라이트 모드에 대응한다.
- Dock에서 숨더라도 앱 번들 아이콘(.icns)은 별도로 필요하다 — Finder, DMG 설치 화면, 시스템 설정 권한 목록, 로그인 항목에 노출된다.
- 아이콘 컨셉(디자인 시안 v1 확정): 형광 옐로우 타일 위 다크 손그림 화살표("잉크 블록"). 화살표는 손맛을 살린다 — 흔들리는 축, 덧그은 자국, 축과 어긋나게 교차하는 화살촉 (시안 2번 글리프로 최종 확정, SVG 원본은 docs/PLAN.md M0 참조). 메뉴바 글리프는 같은 화살표의 단색 템플릿 버전.
- 로그인 시 자동 실행 옵션을 제공한다.
- 창 UI 테마(확정): **"잉크 & 종이"** — 웜 페이퍼 배경(#FAF8F2) + 잉크 텍스트(#1C1E24) + 형광 옐로우 포인트(#FFD400, 하이라이트·포커스 링). 다크 모드는 잉크 배경으로 반전. 프라이머리 버튼은 라이트=잉크 블록+형광 글자, 다크=형광 블록+잉크 글자(앱 아이콘의 반전 관계). UI 크롬에 잉크 5색은 쓰지 않는다(색은 색 차원 전용). 화면 위 요소(마커·커서)는 테마와 무관하게 다크 고정.

## 비목표 (v1 제외)

- 도형·화살표 전용 도구, Shift-직선 (실사용 후 판단으로 보류)
- 커서 하이라이트, 스포트라이트, 화면 확대(줌)
- 그림 저장·불러오기, 텍스트, 개별 요소 편집
- 통과 모드에서 그림 표시 유지 (Epic Pen 방식 — 통과 중 그림은 항상 숨긴다. 버퍼는 유지, 상태 모델 참조)
- 다중 모니터 동시 지원
- 자동 페이드
- 아래 화면 스크롤 추적, 자동 정렬 (원천적으로 불가)

## 제약과 한계

- 오버레이는 화면 좌표에만 그리며 아래 앱의 스크롤을 감지할 수 없다.
- 그림은 정지된 화면에만 유효하다. 통과 모드에서 아래 화면을 움직였다면 돌아왔을 때 이전 그림이 어긋나 있을 수 있다 — 전체 지우기로 비우고 새로 그린다.

## 설계 원칙

- 북극성: 녹화 흐름을 깨지 않는다.
- 기능 판단 기준: 마크면 넣어도 되고, 객체(편집·저장)면 범위 밖이다.
- 추측으로 기능을 넣지 않는다. 실사용이 기능을 끌어당길 때 추가한다.

## 개발 시 주의사항

### 투명 오버레이 창

- transparent, decorations 없음, always-on-top, 화면 전체 덮기로 구성한다.
- macOS에서 웹뷰 배경 투명화는 공개 API가 없어 Tauri의 `macOSPrivateApi` 설정(+ `macos-private-api` feature)이 필요하다. 창 transparent만으로는 웹뷰가 흰 판으로 남는다.
- 그리기 OFF일 때 클릭이 아래 앱으로 통과하도록 cursor 이벤트 무시를 토글한다.
- 전체화면 앱 위에도 표시되도록 `canJoinAllSpaces` + `fullScreenAuxiliary` collection behavior와 창 레벨을 설정하고, Mission Control에 잡히지 않게 한다.
- 창 레벨이 전체화면 앱은 덮되 메뉴바는 덮지 않는지 초기에 검증한다(안전장치 참조).

### Tauri

- OS 밀착 기능(트레이, 단축키, 창 제어)은 Rust 쪽에 둔다.
- React와 Rust는 invoke와 이벤트로 통신한다.
- 웹뷰는 크로미움이 아니라 macOS WebKit이므로 그 기준으로 확인한다.
- 창 제어 플러그인은 capabilities 권한을 명시적으로 허용해야 동작한다.
- LLM은 Tauri v1 코드를 생성하기 쉽다. 공식 llms.txt를 기준으로 v2로 작성한다.

### macOS 권한

- 전역 단축키(RegisterEventHotKey 방식, Tauri global-shortcut 플러그인)는 Accessibility 권한이 필요 없다. 권한이 필요해지는 건 이벤트 탭(CGEventTap)으로 키를 가로챌 때다 — App Store 경로를 열어두기 위해서도 이벤트 탭은 피한다.
- 권한이 필요해지면 사용자가 시스템 설정에서 직접 켜야 하므로 안내 화면을 만든다(온보딩의 조건부 단계).
- TCC 권한은 코드 서명 기준으로 기억된다. 서명 없는 개발 빌드는 리빌드마다 권한이 리셋되므로, 개발 초기부터 일관된 로컬 서명을 쓴다.

### 설정 영속화

- 선택한 색·굵기, 마커 위치, 커스텀 단축키, 온보딩 완료 여부를 저장한다(`tauri-plugin-store` 등).
- Dock 숨김은 Accessory activation policy로 구현하는데, 이 상태에서 일반 창(온보딩·설정)을 띄울 때의 포커스 동작을 확인한다.

### 배포

- 배포와 공증에는 Apple Developer Program(연 99달러)이 필요하다.
- 코드 서명과 공증을 하지 않으면 다른 맥에서 Gatekeeper가 차단한다.
- **v1은 DMG 직접 배포로 확정.** App Store는 웹뷰 투명화가 비공개 API 의존인 데다 샌드박스가 강제라 v1 구조로는 진입 불가. 진입 조건: 오버레이 렌더러를 웹뷰 없는 네이티브 창으로 교체 + 샌드박스 대응. 그 경로를 열어두기 위해 이벤트 탭 기반 해법은 처음부터 피한다(macOS 권한 참조).
- 업데이트는 v1에서 수동(재다운로드)으로 두고, 자동 업데이터(`tauri-plugin-updater`)는 보류한다.
- Apple Silicon과 Intel 양쪽 빌드를 확인한다.

### 그리기 엔진

- 점을 그대로 이으면 각지므로 곡선 보간을 적용한다.
- 캔버스 백킹은 `devicePixelRatio`를 반영한다(Retina에서 흐릿함 방지). 5K 기준 백킹이 5120×2880임을 감안한다.
- 획마다 전체 리렌더하지 않고 증분 드로잉한다.
- coalesced pointer events의 WebKit 지원 여부를 확인해 빠른 손놀림에서 점이 듬성해지지 않게 한다.
- 완료 기준은 "선이 그려진다"가 아니라 "입력 지연이 녹화에 거슬리지 않는다"다. 웹뷰 경유 렌더링 지연이 허용선을 넘으면 네이티브 렌더러 전환(배포 참조)을 앞당긴다.

## 구현 순서

각 단계의 검증 기준을 통과해야 다음으로 간다.

1. Tauri + Vite/React 프로젝트 생성
2. 투명 오버레이 창 띄우기 — 검증: 웹뷰까지 투명(그린 요소만 보임), 전체화면 앱 위 표시, 메뉴바는 안 덮음
3. 마우스로 선 긋기 — 검증: Retina에서 선명, 입력 지연이 녹화에 거슬리지 않음
4. 단축키로 그리기 ON/OFF 토글 (통과 전환) — 검증(최우선 리스크): 그리기 ON에서 ⌘Z가 획을 취소하면서 아래 앱이 활성으로 보임
5. ⌘Z / 전체 지우기, Esc 탈출구 — 검증: 웹뷰를 인위로 멈춰도 Esc로 탈출됨
6. 펜 5색 + 굵기 5단계, 플로팅 마커(색·굵기 변경 + 모드 표시)
7. 메뉴바 상주, Dock 숨김, 종료 항목
8. 첫 실행 온보딩과 단축키 설정(레코더 UI)
9. 권한 처리 (필요해진 경우만)
10. 서명, 공증, DMG 배포
